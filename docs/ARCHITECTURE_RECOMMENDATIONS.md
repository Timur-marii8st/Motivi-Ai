# Архитектурные рекомендации

## Критические исправления (✅ Выполнено)

### 1. ✅ Двойное управление транзакциями
**Проблема**: ToolExecutor делал `commit()` внутри методов, нарушая атомарность.

**Решение**: Заменено на `flush()`. Теперь middleware управляет транзакцией.

**Файлы**: `app/services/tool_executor.py`

### 2. ✅ Обработка ошибок JSON
**Проблема**: ConversationService не обрабатывал markdown JSON блоки.

**Решение**: Добавлен метод `_clean_json()` и обработка ошибок парсинга.

**Файлы**: `app/llm/conversation_service.py`

### 3. ✅ Оптимизация Fact Cleanup
**Проблема**: Self-deduplication имел O(N²) сложность.

**Решение**: Теперь проверяются только эпизоды за последние 24 часа.

**Файлы**: `app/services/fact_cleanup_service.py`

### 4. ✅ Архивирование сырых диалогов
**Проблема**: Важная информация могла теряться при выходе за пределы Redis history.

**Решение**: Добавлен job `archive_raw_conversations_job` для ежедневного архивирования.

**Файлы**: `app/scheduler/jobs.py`

### 5. ✅ Оптимизация Profile Completeness
**Проблема**: Пересчет скора при каждом обращении.

**Решение**: Добавлено кэширование (обновление не чаще раза в час) и возможность передачи pre-loaded объектов.

**Файлы**: `app/services/profile_completeness_service.py`

### 6. ✅ Исправлены индексы для векторов >2000 измерений
**Проблема**: HNSW индексы имеют лимит 2000 измерений, а наши векторы 4096-мерные.

**Решение**: Заменено на IVFFlat индексы, которые поддерживают любую размерность.

**Файлы**: `alembic/versions/20260128_add_vector_indexes.py`

## Рекомендации для будущего развития

### 1. ⚠️ Конкурентность планировщика (Требует изменения инфраструктуры)

**Проблема**: При масштабировании (несколько реплик) каждый процесс запустит свой планировщик.

**Решение**: Вынести планировщик в отдельный сервис.

**Пример docker-compose.yml**:
```yaml
services:
  app:
    # Основное приложение БЕЗ планировщика
    environment:
      - DISABLE_SCHEDULER=true
    deploy:
      replicas: 3  # Можно масштабировать
  
  scheduler:
    # Отдельный контейнер только для планировщика
    build: .
    command: python -m app.scheduler_worker
    environment:
      - SCHEDULER_ONLY=true
    deploy:
      replicas: 1  # Только один экземпляр!
```

**Файлы для создания**:
- `app/scheduler_worker.py` - отдельный entry point для планировщика
- Модифицировать `app/main.py` для поддержки `DISABLE_SCHEDULER`

### 2. ⚠️ Унификация шифрования (Требует миграции данных)

**Проблема**: Используются две библиотеки шифрования (Fernet и Tink).

**Рекомендация**: Перевести все на Tink (более современный стандарт).

**План миграции**:
1. Создать миграцию для `OAuthToken` с использованием `EncryptedTextType`
2. Написать скрипт для перешифрования существующих токенов
3. Удалить `app/utils/encryption.py`
4. Обновить зависимости (убрать `cryptography`)

**Преимущества**:
- Единый стандарт шифрования
- Упрощенная ротация ключей
- Меньше зависимостей

### 3. ⚠️ Удаление легаси CoreMemory (Требует миграции данных)

**Проблема**: Поле `core_text` устарело, но код еще его поддерживает.

**Рекомендация**: Полностью удалить после миграции на продакшене.

**План**:
1. Запустить `scripts/migrate_core_text_to_core_fact.py` на проде
2. Убедиться, что все данные мигрированы
3. Удалить поле `core_text` из модели `CoreMemory`
4. Создать миграцию Alembic для удаления колонки
5. Очистить код от проверок старых полей

### 4. 💡 Оптимизация ReAct цикла

**Текущее состояние**: Системный промпт с контекстом памяти отправляется на каждой итерации.

**Возможная оптимизация** (если провайдер поддерживает):
- Anthropic: использовать prompt caching
- OpenAI: использовать conversation context caching (beta)

**Пример для Anthropic**:
```python
messages = [
    {
        "role": "system",
        "content": system_instruction,
        "cache_control": {"type": "ephemeral"}  # Кэшировать системный промпт
    },
    # ... остальные сообщения
]
```

**Экономия**: До 90% стоимости на повторных вызовах в рамках одной сессии.

   - Для conversation history
