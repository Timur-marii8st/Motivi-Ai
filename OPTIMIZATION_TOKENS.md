# 💰 Оптимизация: Экономия токенов

## Проблема

При исправлении timezone бага изначально было решено передавать timezone через LLM:
- Добавлен параметр `timezone` в tool schema
- Добавлены длинные инструкции в системный промпт
- LLM должен был извлекать timezone из контекста и передавать его

**Проблема**: Это тратит токены на каждый запрос!

## Решение

Timezone берется из профиля пользователя **в коде**, а не через LLM.

### Было (неоптимально):

```python
# Tool schema
"timezone": {
    "type": "string", 
    "description": "REQUIRED: User's IANA timezone name (e.g., Europe/Moscow). Get this from the user's profile..."
}
"required": ["message_text", "reminder_datetime_iso", "timezone"]

# Системный промпт
- ⚠️ ВАЖНО: Когда пользователь говорит время...
- ВСЕГДА передавай timezone пользователя из контекста (user_profile.timezone)
- Время в reminder_datetime_iso должно быть в локальном времени...
- Пример: Если пользователь в Москве (Europe/Moscow) говорит "напомни в 9:30"...

# LLM передает
{
    "timezone": "Europe/Moscow"  // 5 токенов на каждый вызов
}
```

### Стало (оптимизировано):

```python
# Tool schema (короче!)
"description": "Schedule a one-off motivational reminder message for the user at a specific datetime. The time should be in user's local timezone."
# Параметр timezone УДАЛЕН

# Системный промпт (1 строка вместо 5!)
- **schedule_reminder**: Запланировать разовое напоминание для пользователя на конкретную дату и время в его локальном времени

# Код берет timezone сам
user = await self.session.get(User, user_id)
tzname = user.user_timezone if user and user.user_timezone else "UTC"
```

## Экономия

### На один запрос с напоминанием:

| Компонент | Было | Стало | Экономия |
|-----------|------|-------|----------|
| Tool schema | ~150 токенов | ~50 токенов | **100 токенов** |
| Системный промпт | ~100 токенов | ~20 токенов | **80 токенов** |
| LLM output | ~5 токенов | 0 токенов | **5 токенов** |
| **ИТОГО** | **~255 токенов** | **~70 токенов** | **~185 токенов** |

### Годовая экономия:

Предположим:
- 100 напоминаний в день
- 365 дней в году
- Модель: GPT-4 ($0.03 за 1K input tokens)

**Расчет**:
```
185 токенов × 100 напоминаний × 365 дней = 6,752,500 токенов/год
6,752,500 / 1000 × $0.03 = $202.58 экономии в год
```

Для более дешевых моделей (GPT-3.5, Claude Haiku):
- **~$50-100 экономии в год**

## Дополнительные преимущества

### 1. Надежность
- ✅ Timezone всегда актуальный из профиля
- ✅ Нет риска, что LLM передаст неправильный timezone
- ✅ Нет риска, что LLM забудет передать timezone

### 2. Простота
- ✅ Меньше параметров для LLM
- ✅ Проще системный промпт
- ✅ Меньше вероятность ошибок

### 3. Производительность
- ✅ Меньше токенов = быстрее ответ
- ✅ Меньше нагрузка на LLM API

## Применение принципа

**Правило**: Если информация доступна в коде (профиль пользователя, настройки, контекст), не заставляйте LLM передавать её!

### Другие возможности оптимизации:

1. **User ID**: Не нужно передавать через LLM, есть в контексте запроса
2. **Chat ID**: Аналогично
3. **Current time**: Можно брать в коде, не передавать через LLM
4. **User preferences**: Брать из БД, не из контекста LLM

### Когда LLM ДОЛЖЕН передавать параметры:

- ✅ Информация из сообщения пользователя (текст напоминания, время)
- ✅ Решения на основе контекста (какой план создать)
- ✅ Извлеченные данные (факты, намерения)

### Когда LLM НЕ должен передавать:

- ❌ Данные из профиля пользователя (timezone, имя, настройки)
- ❌ Технические ID (user_id, chat_id)
- ❌ Системная информация (текущее время сервера)

## Заключение

Простое изменение архитектуры (timezone из кода вместо LLM) дает:
- 💰 **Экономию $50-200/год**
- 🚀 **Быстрее ответы**
- 🛡️ **Надежнее работа**
- 🧹 **Чище код**

**Принцип**: Не тратьте токены на то, что можно сделать в коде!
